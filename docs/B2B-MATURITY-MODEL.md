# Уровни зрелости прокси-инфраструктуры для B2B

Модель помогает выбрать не «самый сложный» вариант, а минимальный уровень, который даёт компании управляемый и измеримый результат.

## Уровень 1 — подключение

Подходит для проверки учётных данных, маршрута и внешнего IP.

Обязательный результат:

- запрос проходит через прокси;
- ошибки DNS, TLS, timeout и `407` различимы;
- секреты не попадают в вывод;
- есть ожидаемый ответ от разрешённого контрольного endpoint.

Артефакт: [`quickstarts/curl`](../quickstarts/curl/).

Переходите на следующий уровень, когда запрос становится частью приложения или выполняется регулярно.

## Уровень 2 — интеграция приложения

Подходит для одного сервиса или задачи с контролируемой нагрузкой.

Обязательный результат:

- раздельные connect/read/write/pool timeout;
- ограниченные повторные попытки только для безопасных идемпотентных запросов;
- exponential backoff с jitter;
- структурированный результат и стабильные exit codes;
- конфигурация через переменные окружения;
- тесты не требуют реального прокси.

Артефакты:
[`integrations/python-production`](../integrations/python-production/) для
Python и [`integrations/node-production`](../integrations/node-production/) для
Node.js.

Переходите на следующий уровень, когда появляются несколько маршрутов, регионов, команд или требования к доступности.

## Уровень 3 — эксплуатация пула

Подходит для production-нагрузки и общего прокси-контура компании.

Обязательный результат:

- периодические health checks;
- success rate и latency по маршрутам;
- наблюдаемый внешний IP и сводка ротации;
- состояния `healthy`, `degraded`, `failed`;
- отчёт в JSON для CI, мониторинга и аудита;
- retry budget и ограничение параллелизма;
- отсутствие секретов в логах и отчётах.

Артефакт: [`tools/proxy-healthcheck`](../tools/proxy-healthcheck/).

Переходите на следующий уровень, когда инфраструктура обслуживает несколько подразделений, критические процессы или регулируемые данные.

## Уровень 4 — управление B2B-контуром

Подходит для общей корпоративной платформы.

Минимальные организационные требования:

- владелец сервиса и владелец каждого бизнес-сценария;
- каталог разрешённых целей и назначений;
- секреты в корпоративном secret manager;
- разделение production, staging и разработки;
- журнал изменений конфигурации;
- лимиты затрат и квоты команд;
- SLO, алерты и инструкция реагирования на инциденты;
- регулярная проверка договоров, применимого законодательства и правил целевых ресурсов.

Рекомендуемые показатели:

| Показатель | Что измеряет |
|---|---|
| Success rate | Доля успешных разрешённых запросов |
| p50/p95 latency | Типичная и высокая задержка |
| Proxy auth failures | Ошибки конфигурации и истёкшие credentials |
| Distinct observed IP ratio | Фактическую ротацию, если она ожидается |
| Retry amplification | Дополнительную нагрузку от повторных попыток |
| Cost per successful operation | Экономику бизнес-процесса |
| Policy violations | Попытки обратиться к неразрешённым целям |

## Правило выбора

Если компания не может объяснить, кто отвечает за сценарий, какие цели разрешены и как измеряется успешность, масштабировать трафик рано. Сначала закрепите владельца, политику и контрольные показатели.
