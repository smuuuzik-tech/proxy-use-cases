# Шаблон SLO и runbook для прокси-контура

Значения ниже — структура для обсуждения, а не универсальные нормативы. Цели должны следовать из критичности конкретного бизнес-процесса и подтверждаться измерениями.

## Карточка сервиса

| Поле | Значение |
|---|---|
| Владелец бизнес-сценария | `<team-or-role>` |
| Технический владелец | `<team-or-role>` |
| Назначение | `<approved-purpose>` |
| Разрешённые endpoints | `<allowlist-reference>` |
| Регионы | `<regions>` |
| Окно обслуживания | `<24x7 or business-hours>` |
| Канал инцидентов | `<incident-channel>` |

## SLI

### Успешность

```text
success_rate = successful_allowed_operations / all_allowed_operations
```

Не смешивайте в один показатель:

- ошибки приложения;
- proxy authentication;
- transport timeout;
- HTTP-ответ целевого endpoint;
- блокировки внутренней policy.

### Задержка

Измеряйте минимум p50 и p95 полного бизнес-запроса. Отдельно полезно хранить время соединения и ожидания ответа.

### Ротация

Используется только для маршрутов, где ротация является частью контракта:

```text
distinct_ip_ratio = distinct_observed_ips / successful_checks
```

Показатель не доказывает качество сам по себе. Его нужно оценивать вместе с размером выборки, длительностью окна и политикой сессий.

### Retry amplification

```text
retry_amplification = total_attempts / original_operations
```

Рост показателя часто означает деградацию маршрута или целевого endpoint и может резко увеличить стоимость.

## Шаблон целей

| SLI | Цель | Окно | Исключения |
|---|---:|---|---|
| Success rate | `<target>` | `<rolling window>` | `<documented exclusions>` |
| p95 latency | `<threshold>` | `<rolling window>` | `<documented exclusions>` |
| Auth failures | `0` | `<window>` | Плановая ротация credentials |
| Retry amplification | `<maximum>` | `<window>` | Согласованные нагрузочные тесты |

## Состояния healthcheck

- `healthy`: все обязательные пороги выполнены;
- `degraded`: запросы проходят, но нарушен хотя бы один предупредительный порог;
- `failed`: обязательный endpoint недоступен, аутентификация не проходит или превышен критический порог.

Точные пороги хранятся рядом с конфигурацией сервиса и проходят code review.

## Runbook

### 1. Определить класс ошибки

Проверьте структурированный отчёт и отделите:

1. configuration/auth;
2. DNS/connect/TLS;
3. timeout;
4. HTTP target;
5. policy;
6. локальную ошибку приложения.

### 2. Ограничить влияние

- остановить автоматическое увеличение параллелизма;
- не повышать число retries без анализа;
- переключить только заранее проверенный резервный маршрут;
- сохранить отчёт без credentials и клиентских данных.

### 3. Проверить контрольным инструментом

Сначала Level 1 quickstart, затем `proxy-healthcheck`. Используйте только согласованный контрольный endpoint.

### 4. Восстановить

- исправить конфигурацию или credentials через утверждённый процесс;
- проверить один маршрут;
- проверить малую выборку;
- постепенно восстановить нагрузку;
- убедиться, что retry amplification вернулся к норме.

### 5. Зафиксировать

В отчёте об инциденте укажите временную шкалу, класс ошибки, затронутые процессы, изменения конфигурации и действие, предотвращающее повторение. Не включайте секреты и полные proxy URL.
