# План развития GitHub на 90 дней

## Задача системы

GitHub должен одновременно выполнять три функции:

1. давать B2B-команде запускаемый ответ на конкретную инженерную задачу;
2. показывать подход Андрея через код, тесты, архитектуру и документацию;
3. переводить заинтересованного читателя на
   [экспертную страницу](https://andrey-proxy-advisor-private.vercel.app/github?utm_source=github&utm_medium=roadmap&utm_campaign=proxy_use_cases),
   где можно описать задачу или выбрать время звонка.

Основная воронка:

`поисковый запрос → решение на GitHub → документация/код → сайт → консультация`.

## Редакционные и инженерные правила

- Сначала B2B-сценарий и измеримый результат, затем технология.
- Основные решения остаются провайдер-независимыми.
- Особенности конкретного API живут в отдельном `vendor-specific` адаптере.
- У каждого решения есть быстрый старт, офлайн-тесты, безопасные значения по
  умолчанию и раздел «когда не использовать».
- Клиентские данные, реальные credentials и закрытые конфигурации не
  публикуются.
- Новый самостоятельный репозиторий появляется только при отдельном жизненном
  цикле, пользователях и владельце поддержки.

## Первые 14 дней: фундамент и измерение

### Публикация

- Связать профиль, основной README и сайт взаимными ссылками с UTM-метками.
- Добавить roadmap, шаблоны issue и форму запроса нового B2B-сценария.
- Выпустить первый тегированный релиз текущей базы.
- Для каждого решения добавить короткий блок: задача, входные данные, результат,
  эксплуатационные ограничения, следующий уровень.

### Измерение

Сначала фиксируется двухнедельный baseline:

- переходы `GitHub → сайт`;
- клики `сайт → запись на звонок`;
- уникальные клоны и успешные локальные запуски;
- вопросы и issues по конкретным решениям;
- квалифицированные B2B-обращения, в которых посетитель называет сценарий,
  объём или текущую проблему.

Звёзды учитываются как сигнал узнаваемости, но не как основной результат.

## Дни 61–90: дистрибуция и конверсия

Каждая законченная тема проходит один и тот же цикл:

1. запускаемое решение и README на GitHub;
2. подробный разбор в блоге;
3. короткая версия для LinkedIn и TenChat;
4. страница B2B-сценария на сайте;
5. CTA на диагностику именно этого сценария;
6. ответы на вопросы превращаются в FAQ, issue или следующее улучшение кода.

Рекомендуемый ритм — один законченный инженерный релиз раз в две недели и
два–три производных материала из него. Контент не создаётся отдельно от
продукта: один проверенный артефакт питает все каналы.

## Архитектура репозиториев

На ближайшие 90 дней основной код остаётся в `proxy-use-cases`: так проще
искать решения по уровням и поддерживать единый стандарт качества.

Кандидаты на выделение позже:

- `proxy-healthcheck` — если появляется отдельный CLI-релиз и пользователи;
- `proxy-policy-gateway` — если появляется самостоятельный сервис и deployment;
- `proxy-observability` — если адаптеры метрик начинают жить независимо.

Исходный код личного сайта не входит в публичную GitHub-архитектуру.

## Контрольная панель раз в две недели

| Слой | Метрика | Вопрос |
|---|---|---|
| Охват | Показы страниц GitHub и поисковые переходы | На какие задачи нас находят? |
| Активация | Клоны, запуски, прохождение тестов | Решение действительно можно применить? |
| Доверие | Повторные посетители, issues, ссылки на материалы | Подход считают полезным и проверяемым? |
| Переход | CTR из README на сайт | Код вызывает интерес к разбору задачи? |
| Бизнес | Квалифицированные обращения и звонки | Приходит нужная B2B-аудитория? |

После baseline для каждой метрики устанавливается квартальная цель. Цель
пересматривается по качеству обращений, а не по максимальному объёму трафика.
